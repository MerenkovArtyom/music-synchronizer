# Storyline

## 2026-04-24

### Commit 1: project skeleton

- Создан дизайн проекта для синхронизации Yandex Music -> Obsidian.
- Подготовлен implementation plan для этапа каркаса проекта.
- Начата реализация skeleton-only этапа.
- Добавлены `pyproject.toml`, `.gitignore`, `README.md` и базовая структура `src/`.
- Добавлены `.env.example`, `Settings` и CLI-команды `show-config`/`sync`.
- Добавлены и пройдены базовые тесты для конфигурации и CLI.
- Подготовлен локальный `.env` с путём к Obsidian-хранилищу.

### Commit 2: yandex likes client and track model

- Добавлена зависимость `yandex-music` для получения `Любимых треков`.
- Добавлены `TrackInfo` и адаптер `YandexMusicClient` с нормализацией liked tracks во внутреннюю модель.
- Покрыта тестами нормализация ответа API и поведение при отсутствии зависимости.
- Добавлен `ObsidianExporter`, который записывает отдельные markdown-файлы в `tracks/` с именами по названию песни.
- Для конфликтов имён добавлены правила уникализации, а `track_id` сохранён как стабильный идентификатор в метаданных.
- Формат заметок унифицирован: YAML frontmatter плюс короткое markdown-тело.
- Добавлены тесты на генерацию трековых заметок, переименование, архивирование и восстановление структуры.
- Добавлен `SyncService` и подключен к CLI-команде `music-sync sync`.
- Реализовано архивирование удалённых треков в `tracks/_removed/`, восстановление при повторном появлении и удаление устаревшего `playlist.md`.
- Обновлён `README.md` под фактическое поведение синхронизации без отдельного индексного файла.
- Добавлены CLI-тесты на успешный sync и безопасную обработку ошибок без частичной записи файлов.

## 2026-04-25

### Commit 3: track tags in sync and notes

- В `TrackInfo` добавлено поле `tags` для хранения тегов трека во внутренней модели.
- `YandexMusicClient` теперь подтягивает жанровые теги из доступных метаданных трека.
- `ObsidianExporter` записывает теги в frontmatter заметок и сохраняет вручную добавленные теги при повторной синхронизации.
- Изначально формат заметки использовал единое поле `tags` без разделения на синхронизированные и ручные значения.
- Добавлены тесты на нормализацию тегов, запись в заметки и объединение ручных тегов с синхронизированными.

### Commit 4: list saved tracks by tag

- Добавлена CLI-команда `music-sync list --tag <tag>` для вывода активных сохранённых треков из Obsidian.
- Фильтрация по тегу работает по точному совпадению без учёта регистра и не включает заметки из `tracks/_removed/`.

## 2026-04-27

### Commit 6: split synced and manual tags

- `ObsidianExporter` разделяет теги заметки на `system_tags` и `user_tags`.
- Во время `sync` обновляются только `system_tags`, а `user_tags` перечитываются из существующей заметки и не теряются.
- Для старых заметок с единым `tags` добавлена мягкая миграция: legacy-теги трактуются как `user_tags` и переписываются в новый формат при следующем sync.
- `music-sync list --tag` теперь ищет по объединению `system_tags` и `user_tags`, сохраняя прежнюю удобную семантику команды.
- Вывод команды печатает строки в формате `Название - Artist1, Artist2`, а для пустого результата показывает отдельное сообщение.
- Обновлены `README.md` и CLI-тесты под новый сценарий списка по тегу.

### Commit 5: list saved tracks by artist

- CLI-команда `music-sync list` расширена новым фильтром `--artist <artist>` по аналогии с `--tag`.
- Поиск по артисту работает по точному совпадению без учёта регистра и учитывает только активные заметки из `tracks/*.md`.
- Для `list` закреплено правило взаимоисключающих фильтров: нужно передать ровно один из `--tag` или `--artist`.
- Обновлены `README.md` и CLI-тесты под сценарии списка по артисту и валидацию нового контракта команды.
